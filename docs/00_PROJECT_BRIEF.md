# 项目背景、目标与创新点说明

> 本文档用于帮助 Codex 和项目成员快速理解本项目的背景、痛点、目标、创新点和当前 Demo 阶段边界。后续开发前应先阅读本文档，再阅读模块设计、Maven 模块规划、数据模型和开发计划文档。

---

## 1. 项目名称

**中文名称：** 基于多智能体协同的网络意图翻译与闭环验证系统

**英文简称：** MAC-TAV

**英文全称：** Multi-Agent Collaborative Translation And Verification

---

## 2. 项目一句话简介

本项目面向网络实验教学与轻量网络方案验证场景，构建一个基于多智能体协同的网络意图处理系统，使用户可以通过自然语言描述网络需求，由系统自动完成意图解析、网络方案规划、配置生成、执行适配、验证评估和结果展示，形成“意图 -> 规划 -> 配置 -> 执行 -> 验证”的闭环流程。

---

## 3. 赛题背景理解

本项目面向“智能体互联网创新攻关”赛项。赛项关注智能体互联网、大模型智能体、多智能体协作、网络意图自动翻译、闭环验证、网络策略自适应生成、MCP/A2A 协同等方向。

传统网络系统主要围绕设备、链路、协议和配置命令展开，用户需要具备较强的网络基础，才能把业务需求翻译成底层设备配置。随着大模型、智能体和工具调用技术的发展，网络系统可以逐步从“人工配置驱动”转向“意图驱动”和“智能体协同驱动”。

本项目选择“网络意图翻译与闭环验证”作为切入点。目标不是简单生成配置命令，而是让系统能够理解用户业务目标，规划网络方案，生成结构化配置，并通过规则验证或仿真验证判断方案是否真正满足用户意图。

## 4. 项目背景

在网络实验教学和轻量网络运维验证场景中，用户通常可以用自然语言描述自己的网络目标，例如：

> 构建一个办公区和访客区隔离的网络，两个区域都能访问互联网，访客区不能访问服务器，采用 OSPF。

但是，要真正落地这个需求，用户仍然需要完成大量专业工作：

1. 判断需要哪些网络设备。
2. 设计路由器、交换机、主机之间的拓扑关系。
3. 规划 VLAN、IP 地址、网关和路由协议。
4. 编写交换机、路由器、防火墙或 SDN 控制器配置。
5. 手动执行配置命令或仿真环境命令。
6. 使用 ping、traceroute、流表查询等方式验证结果。
7. 如果验证失败，还需要分析是拓扑问题、地址问题、路由问题、ACL 问题，还是执行环境问题。

这个过程对初学者和轻量运维场景都不友好，容易出现配置错误、验证遗漏和排障困难等问题。

因此，本项目希望通过多智能体协同、统一网络模型、结构化配置生成、执行适配和可视化展示，降低网络配置和验证门槛，让用户能够从自然语言意图出发，得到可解释、可验证、可追踪的网络方案。

真实大模型、真实 RAG、真实 Mininet/Ryu、MySQL、Redis、Qdrant 都属于后续阶段接入能力，不是当前 Demo 的强制依赖。

## 5. 目标场景

### 5.1 网络实验教学场景

适合计算机网络、网络工程、SDN、路由交换等课程实验。学生可以通过自然语言描述实验目标，系统自动生成拓扑、配置和验证结果，帮助学生理解“需求如何转化为网络配置”。

### 5.2 轻量网络方案验证场景

适合对小型企业网络、实验网络、临时网络方案进行快速设计和验证。例如办公区、访客区、服务器区之间的访问控制、VLAN 隔离、路由互通和公网访问验证。

### 5.3 多智能体协同 Demo 场景

适合作为比赛演示系统，展示多个智能体如何围绕同一个网络任务协同工作，包括意图理解、方案规划、配置生成、执行适配和结果验证。

## 6. 现有痛点

### 6.1 从业务需求到设备配置的转换门槛高

用户的真实需求通常是业务语言，例如“访客区不能访问服务器”或“办公区可以访问公网”。而网络设备需要的是 VLAN、IP、路由、ACL、NAT、流表等底层配置。中间的转换过程依赖网络经验，初学者难以独立完成。

### 6.2 手工配置响应慢、容易出错

传统方式需要人工完成拓扑规划、配置编写、命令下发和验证测试。配置命令较多时，容易出现接口写错、网段冲突、VLAN 漏配、ACL 方向错误、路由缺失等问题。

### 6.3 只生成配置不等于满足意图

很多工具或大模型只能生成配置文本，但无法证明这些配置是否真的满足用户目标。例如生成了 ACL，并不代表访客区真的无法访问服务器；生成了路由配置，也不代表网络真的可达。

### 6.4 缺少闭环验证和错误反馈

传统配置流程往往停留在“生成配置”阶段，缺少“验证配置是否达成意图”的闭环机制。当网络不通或策略失效时，用户需要自己定位错误原因。

### 6.5 可解释性不足

用户不仅需要最终配置，还需要知道：系统为什么这样规划？某段配置对应哪条用户意图？验证失败时问题出在哪里？如果没有可解释链路，系统生成结果难以让用户信任。

### 6.6 仿真环境和真实网络工具割裂

网络教学和验证中常用 Mininet、Ryu、eNSP、Docker 等工具，但这些工具通常需要手动搭建和操作。用户需要学习额外命令，才能将网络方案跑起来并观察结果。

### 6.7 单智能体处理复杂任务容易混乱

网络意图处理包含语义理解、方案规划、配置生成、执行适配、结果验证等多个阶段。如果全部交给一个智能体完成，容易出现职责混乱、输出不稳定、错误难定位的问题。

## 7. 项目目标

本项目的总体目标是实现一个基于多智能体协同的网络意图翻译与闭环验证系统。

系统应支持用户输入自然语言网络需求，并由多个模块协同完成：

1. **意图解析：** 将自然语言需求转换为结构化 `NetworkIntent`。
2. **网络规划：** 根据业务意图生成 `NetworkPlan`，包括拓扑、区域、IP、VLAN、路由和安全策略。
3. **配置生成：** 根据 `NetworkPlan` 生成 `ConfigSet`，包括设备配置命令、配置块解释和回滚信息。
4. **执行适配：** 将 `NetworkPlan + ConfigSet` 转换为 Mininet/Ryu/DryRun 可执行内容。
5. **验证评估：** 根据测试结果判断网络方案是否满足用户意图，输出 `ValidationReport`。
6. **过程展示：** 展示意图、拓扑、配置、执行结果、验证报告和 Agent 执行轨迹。
7. **后期自愈：** 当验证失败时，进一步分析失败原因并生成修复建议或重新规划方案。

---

## 8. 项目核心流程

系统主流程如下：

```text
用户输入自然语言网络需求
        ↓
Intent Agent 解析业务意图，输出 NetworkIntent
        ↓
Planning Agent 生成网络方案，输出 NetworkPlan
        ↓
Configuration Agent 生成设备配置，输出 ConfigSet
        ↓
Execution Module 转换并执行 DryRun / 仿真任务，输出 ExecutionReport
        ↓
Verification Agent 判断意图是否达成，输出 ValidationReport
        ↓
Web / Visualization 展示拓扑、配置、执行过程和验证结果
```

后期加入 Healing Agent 后，系统将形成更完整的闭环：

```text
验证失败
  ↓
Healing Agent 分析失败原因
  ↓
判断修复类型
  ↓
重新规划 / 重新生成配置 / 重新执行 / 向用户澄清
  ↓
再次验证
```

## 9. 项目核心设计思想

### 9.1 意图驱动，而不是命令驱动

系统从用户业务目标出发，而不是直接让用户输入设备命令。用户描述“想实现什么”，系统负责推导“网络应该怎么设计”和“命令应该怎么生成”。

### 9.2 多智能体分工协作

系统将复杂任务拆分给不同智能体或模块：

1. Intent Agent：理解用户需求。
2. Planning Agent：设计网络方案。
3. Configuration Agent：生成配置命令。
4. Execution Module：执行仿真适配。
5. Verification Agent：判断是否满足意图。
6. Healing Agent：后期负责故障诊断与修复。

这种分工可以减少单智能体处理复杂任务时的混乱，提高系统可维护性和可解释性。

### 9.3 统一网络模型作为数据底座

系统通过 Model Core / NetworkWorkspace 管理各阶段产物，包括 `NetworkIntent`、`NetworkPlan`、`ConfigSet`、`ExecutionReport`、`ValidationReport` 等，保证不同模块之间的数据一致性、版本可追踪和过程可回放。

### 9.4 配置生成必须结构化

配置生成模块不只返回一整段命令文本，而是按设备、配置块、解释、回滚命令和追溯关系组织 `ConfigSet`。这样后续才能支持展示、验证、定位和自愈。

### 9.5 执行适配必须隔离环境差异

Execution Module 不能直接执行 Huawei CLI，而应通过 Execution Adapter 将 `NetworkPlan + ConfigSet` 转换为 Mininet/Ryu/DryRun 可执行内容。当前 Demo 优先使用 DryRun / Mock 结果。

### 9.6 验证是核心，不是附加功能

系统不以“生成配置”为终点，而是以“验证用户意图是否达成”为闭环目标。Verification Agent 需要根据 `NetworkIntent` 和 `TestResult` 判断连通性、隔离性、路由策略和安全策略是否符合预期。

### 9.7 前端展示全过程

系统不仅展示最终结果，还展示每个阶段的中间产物和 Agent 执行轨迹，让用户知道系统如何理解需求、如何规划网络、如何生成配置、如何执行验证。

## 10. 项目创新点

### 10.1 从自然语言意图到网络方案的自动转化

系统支持用户以自然语言描述网络目标，并通过 Intent Agent 与 Planning Agent 将业务意图转化为结构化网络方案，降低用户使用门槛。

### 10.2 多智能体协同的网络意图处理流程

系统通过多个 Agent 分工协作，将意图理解、网络规划、配置生成、验证评估等任务拆分处理，避免单一模型承担全部复杂逻辑。

### 10.3 统一 NetworkWorkspace 状态中心

系统通过 Model Core / NetworkWorkspace 保存各阶段产物、版本关系、执行状态和追溯关系，使整个流程具备可查询、可追踪、可恢复的能力。

### 10.4 配置块级可解释生成

Configuration Agent 生成 `ConfigSet` 时，按设备和 `commandBlocks` 组织配置，保留每段配置的解释、依赖关系、回滚命令和关联意图，增强可解释性和可维护性。

### 10.5 规则验证与仿真验证结合

系统可以先通过规则验证快速判断网络方案是否存在明显问题，再通过 Mininet/Ryu 或 Mock 执行环境进行动态测试，形成更可信的闭环验证。

### 10.6 执行适配层屏蔽仿真环境差异

Execution Module 通过 Execution Adapter 将 `NetworkPlan + ConfigSet` 转换为 Mininet 拓扑脚本、Linux 主机命令、Ryu 流表规则或 DryRun 测试结果。这样系统不会直接把传统设备 CLI 强行输入 Mininet，而是进行环境适配。

### 10.7 可视化展示意图到验证的全过程

前端不仅展示拓扑和配置，还展示任务进度、Agent 执行过程、验证结果和失败原因，使复杂网络逻辑更加直观。

## 14. 与普通配置生成工具的区别

普通配置生成工具通常只解决：

```text
输入需求 -> 输出配置命令
```

本项目希望解决：

```text
输入业务意图
  ↓
结构化理解
  ↓
网络方案规划
  ↓
配置命令生成
  ↓
执行与仿真适配
  ↓
意图验证
  ↓
可解释展示
  ↓
后期自愈优化
```

因此，本项目的重点不是“生成一段命令”，而是“让网络意图处理具备可规划、可执行、可验证、可解释、可追踪的闭环能力”。

## 15. 面向 Codex 的开发理解要求

Codex 在开发本项目时，需要遵守以下理解：

1. 本项目当前是 Demo 阶段，优先保证流程跑通。
2. 当前阶段只做 Maven 多模块单体。
3. 不要一开始接入真实大模型、真实 RAG、真实 Mininet/Ryu。
4. 不要强制引入 MySQL、Redis、Qdrant。
5. 不要创建真实微服务，不引入服务注册发现。
6. 不要把所有代码写到 `mac-tav-web`。
7. 只有 `mac-tav-web` 是 Spring Boot 启动模块。
8. 所有共享 DTO 放在 `mac-tav-model`。
9. 所有 Agent 通用抽象放在 `mac-tav-agent-core`。
10. Model Core 不调用大模型，不生成配置，不执行仿真，只负责状态和产物管理。
11. Execution Module 不应直接执行 Huawei CLI，而应通过 Execution Adapter 转换为目标环境可执行内容。
12. Verification Agent 负责判断测试结果是否满足用户意图。
13. Healing Agent 后期扩展，当前阶段只保留 TODO 或 Mock 说明。

## 16. 项目价值总结

本项目的价值在于把传统网络配置流程从“人工命令驱动”提升为“自然语言意图驱动”，并通过多智能体协同、统一网络模型、结构化配置生成和闭环验证机制，让网络设计、配置和验证过程更加自动化、可解释和可展示。

对于比赛演示而言，本项目能够体现：

1. 大模型与网络技术融合的后续扩展空间。
2. 多智能体任务协同。
3. 网络意图自动翻译。
4. 网络策略自动生成。
5. 规则验证和仿真验证。
6. 可视化 GUI 展示。
7. 后续自愈闭环扩展能力。
